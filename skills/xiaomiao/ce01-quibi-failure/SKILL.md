# 反例: Quibi的18亿美元失败

## R — 原文 (Reading)

> Quibi在2020年4月倒闭，结束了为期6个月的生命。它筹集了18亿美元，拥有好莱坞最大牌的名字，但每天只有30万用户——而Disney+在同一天有400万用户。

> — 萨希尔·拉文吉亚, 第四章

---

## I — 方法论骨架 (Interpretation)

这个反例揭示了**超额融资如何掩盖产品-市场契合(PMF)的缺失**。

核心逻辑链:
1. 充足资金 → 雇佣豪华团队 → 大规模营销 → 跳过PMF验证
2. 跳过PMF验证 → 产品无法留住用户 → 用户流失
3. 用户流失 → 收入无法覆盖成本 → 死亡

反直觉教训: 钱越多，越危险。因为钱给你一种"可以跳过验证"的幻觉，让你跳过最关键的"客户真的愿意付钱吗"这个问题。

---

## A1 — 书中的应用 (Past Application)

### 案例 1: Quibi的18亿美元失败
- **问题**: Quibi有顶级内容(Hollywood明星)、充足资金(18亿)、精美产品,但没有用户
- **方法论的使用**: 验证"大制作+大资金=成功"这个假设
- **结论**: 内容和资金都无法替代PMF——用户第一天下载≠真正需要这个产品
- **结果**: 6个月后公司倒闭

### 案例 2: Gumroad的教训对比
- **问题**: 萨希尔从追逐独角兽到追求盈利
- **方法论的使用**: 用"先有100个客户"验证PMF，而不是用资金验证
- **结论**: "我在追逐独角兽...我基本上不得不从头开始建立整个团队"
- **结果**: 收缩后实现盈利

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 用户在讨论"我们应该融一大笔钱来做大"——暗示跳过早期验证
2. 用户说"大张旗鼓的发布会"是增长策略——把营销当救命稻草
3. 用户用"XX亿估值"、"XX万美元融资"衡量成功
4. 用户相信"钱多总比钱少好"——认为资金充足可以弥补产品缺陷
5. 用户说"等我们融到钱就招人/扩张/做营销"

- "大厂都这么做"
- "钱多了自然增长"

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **验证产品-市场契合先于讨论融资**
   - 完成标准: 用户能说出"谁愿意付钱"的具体答案（不是"大市场"）

2. **检查是否在用资金代替验证**
   - 完成标准: 如果用户说"融到钱就..."，追问"你的100个付费客户是谁"
   - 判停条件: 若用户已有100个自愿付费的真实客户，可跳过此skill

3. **强调"验证优先"原则**
   - 完成标准: 用户理解了"钱多跳过验证比钱少更危险"

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 已验证PMF后讨论合理扩张——这是正常的规模化，不是本skill的适用场景
- 传统企业融资/项目融资——本skill专属于创业场景
- 讨论资本运作/投资策略——本skill关注创业公司内部决策

### 作者在书中警告的失败模式

- 筹集巨额资金后追求快速扩张，而不是先验证PMF
- 依赖"大张旗鼓的发布会"而非有机增长
- 产品无法留住用户（第一天下载量≠真实需求）

### 如何避免 Quibi 式的失败

1. **永远先问: 谁愿意付钱** — 在融资之前，先找到100个愿意付费的真实客户
2. **用小资金验证** — 如果1万美元无法验证，1000万美元也不会有帮助
3. **把"大发布会"换成"小口碑"** — 3个客户主动推荐 > 1000个下载后卸载
4. **测试留存率** — 30天后的用户留存比第一天下载量更能说明问题

### 容易混淆的邻近方法论

- "精益创业"的MVP——本skill强调的是人工验证而非代码MVP
- "增长黑客"——本skill警告的是跳过PMF的增长
