# 反例: 过度融资陷阱

## R — 原文 (Reading)

> 你可以看到有1000万、2000万、5000万美元的公司为了追求增长而在招聘和基础设施上过度投入...然后他们就倒下了。

> — 萨希尔·拉文吉亚, 第六章

---

## I — 方法论骨架 (Interpretation)

这个反例揭示了**钱多导致的过度扩张比钱少导致的保守更危险**。

核心逻辑链:
1. 钱多 → 招聘多 → 成本高
2. 成本高 → 需要更多钱
3. 需要更多钱 → 失去自主权
4. 失去自主权 → 决策变形
5. 决策变形 → 失去方向

反直觉教训: 保守最多让你活不长，但过度扩张会让你在快速烧钱中失去方向。

## A1 — 书中的应用 (Past Application)

### 案例 1: Gumroad创始人薪资演变
- **问题**: 融资后的薪资设定
- **方法论的使用**: 创始人薪资始终与公司状态挂钩
- **结论**: 从36000到12万美元，始终匹配公司健康度
- **结果**: 保持与公司状态的同步

### 案例 2: Quibi的18亿美元
- **问题**: 超额融资导致无法专注PMF
- **方法论的使用**: 用资金代替产品验证
- **结论**: 18亿美元仍然6个月破产
- **结果**: 快速烧钱，快速死亡

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 用户刚融完钱，计划6个月从14人招到30人——招人速度超过增长速度
2. 用户说"等我们大了就好了"——用未来掩盖当前问题
3. 用户问"这个速度正常吗"——需要人均产出分析
4. 用户用"钱解决问题"而非"优化效率"——用资金代替执行力
5. 用户不关注人均产出——只看总规模

### 语言信号 (用户的话里出现这些就应激活)

- "招人速度超过业务增长速度"
- "员工人均产出下降"
- "等我们大了就好了"
- "用钱解决问题"
- "人多力量大"

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **计算人均产出**
   - 完成标准: 月收入 ÷ 员工数 = 人均产出(每月)

2. **评估招聘速度的合理性**
   - 完成标准: 用户算出招到30人后，月收入需要增长多少才能维持同样效率
   - 判停条件: 若未来6个月收入无法增长2倍，招聘速度已超学习速度

3. **设定"最小有效团队"**
   - 完成标准: 用户理解"14个人能做多少事"比"14个人能做多少新事"更重要

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 已验证PMF后的合理扩张——这是正常的规模化
- 已有盈利能力的公司讨论扩张——这不在本skill范围内
- 特殊行业需要快速扩张（如某些SaaS的PLG场景）——需要case-by-case判断

### 作者在书中警告的失败模式

- 资金充裕导致过度开支
- 跳过"小而美"直接追求"大而全"
- 用钱解决问题而非优化效率

### 如何避免过度融资陷阱

1. **用人均产出衡量** — 如果14个人每月创造10万收入，人均产出约7100/月；招到30人后，月收入需要21万
2. **招聘速度不超过学习速度** — 什么会在未来6个月内让月收入增长2倍？如果答案不清晰，不要招
3. **先止血再扩张** — 先证明你能用14个人创造价值，再考虑30个人
4. **钱多时更要谨慎** — 充裕的资金是最大的危险，因为它让你有错误的自信

### 容易混淆的邻近方法论

- "融资就是为了招人"——本skill强调招人速度需要匹配业务增长速度
- "规模经济"——那是大公司的逻辑，早期公司还没有到那个规模
