# 术语: 有价值的人工流程 (Manual Valuable Process, MVP前置)

## R — 原文 (Reading)

> 我所说的并不是最小化可行产品，而是它的之前的有价值的人工流程，它将是你试图创建的生意的基础。

> — 萨希尔·拉文吉亚, 第三章

---

## I — 方法论骨架 (Interpretation)

MVP前置流程是一种**用人工服务代替代码开发来验证商业假设的方法**。

核心逻辑:
1. 不要先写代码 → 先人工服务客户
2. 用人工方式找出最佳服务方式 → 记录成操作手册
3. 验证客户愿意付费 → 才自动化
4. 自动化的是已被验证的流程 → 而非基于假设的代码

这与精益创业的MVP根本不同:
- 精益MVP从代码开始 → 基于假设
- MVP前流程从人工开始 → 基于真实反馈

---

## A1 — 书中的应用 (Past Application)

### 案例 1: Gumroad手动付款时代
- **问题**: 如何验证人们愿意为在线支付写博客收费
- **方法论的使用**: 萨希尔用PayPal手动每月结算
- **结论**: 用人工方式验证了商业模式
- **结果**: 验证成功后才做产品

### 案例 2: Endcrawl案例
- **问题**: 如何处理电影片尾字幕
- **方法论的使用**: 用Google表格和Perl脚本手动处理
- **结论**: 验证后才做产品
- **结果**: 产品成为电影行业的标准工具

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 用户说"我有一个App想法，想开始写代码"——需要先验证，而非直接开发
2. 用户问"如何验证产品想法"——需要MVP前流程
3. 用户讨论"MVP是不是太早"——需要理解真正的MVP在代码之前
4. 用户说"产品做出来用户自然会有"——需要人工验证先于产品开发
5. 用户有3个产品想法，不知道先做哪个——需要用人工方式验证哪个最痛

### 语言信号 (用户的话里出现这些就应激活)

- "最小可行产品"
- "先做产品再验证"
- "如何验证客户需求"
- "应该先开发还是先调研"
- "产品做出来用户自然会有"

### 与相邻 skill 的区分

- 与 "精益创业MVP" 的区别: 精益从代码开始，本方法从人工开始
- 与 "商业计划/精益画布" 的区别: 画布是纸上验证，本方法是行动验证

---

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **找到你最想解决的问题客户**
   - 完成标准: 用户明确"谁有这个问题"

2. **用人工方式服务他们**
   - 完成标准: 用户亲自用微信/邮件/电话服务5-10个客户

3. **记录最佳服务方式**
   - 完成标准: 用户把服务过程写成步骤清单（这是你的"操作手册"）

4. **验证付费意愿**
   - 完成标准: 在人工服务时就收取费用（哪怕1美元）
   - 判停条件: 若5-10个客户中超过50%愿意付费，自动化才有意义

---

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 已有付费用户需要迭代优化——这时应该用g03流程化
- 纯技术产品无法人工模拟——某些确实需要技术的产品需case-by-case判断
- 用户已有明确的客户需求和验证——跳过此阶段

### 作者在书中警告的失败模式

- 跳过人工验证直接做代码——产品基于假设而非真实反馈
- 用调研代替真实服务——口头承诺不等于付费意愿
- 自动化一个还没被验证的流程——固化错误

### 容易混淆的邻近方法论

- "精益创业MVP" — 那是代码先行，本方法是人工先行
- "商业计划" — 那是纸上推演，本方法是行动验证
