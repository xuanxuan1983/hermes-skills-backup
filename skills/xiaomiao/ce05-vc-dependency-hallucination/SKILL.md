# 反例: 对风投"救兵"的依赖幻觉

## R — 原文 (Reading)

> 创始人不知道答案的原因是他们认为自己不需要知道答案。他们等待着投资人"嗖"的一下飞过来拯救他们。

> — 萨希尔·拉文吉亚, 第六章

---

## I — 方法论骨架 (Interpretation)

这个反例揭示了**投资人的激励结构与创业者的生存需求根本不一致**。

核心逻辑链:
1. 创始人: 需要公司活下去 → 投资人: 需要高倍数回报
2. 投资人需要上行空间 → 你需要的是止血
3. 创始人认为"投资人来了问题就解决"
4. 实际上: 投资人不是来救你的，是来获取上行空间的

反直觉教训: 风投的激励（追求高倍数）≠创始人的激励（活下去）。依赖外部救援是一种认知偏差。

---

## A1 — 书中的应用 (Past Application)

### 案例 1: 依赖幻觉的普遍性
- **问题**: 创始人等待投资人"嗖"的一下飞过来拯救
- **方法论的使用**: 检验"融资是创业正常路径"的假设
- **结论**: 投资人不为糟糕的决策兜底
- **结果**: 等待救兵的公司往往等不到救兵

### 案例 2: Quibi的反面教训
- **问题**: 18亿美元也没能救一个没有PMF的产品
- **方法论的使用**: 用实际案例验证"资金解决一切"
- **结论**: 没有PMF，钱越多死得越快
- **结果**: 6个月后破产

---

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 用户说"等融到B轮就好了"——用融资时间表代替执行计划
2. 用户说"投资人应该帮我们"——把投资人当资源方而非利益方
3. 用户不清楚公司的盈亏平衡点——不关心公司的默认存活状态
4. 用户在账上钱快用完时第一反应是"找投资人"而非"止血"
5. 用户说"我有投资人关系"——把关系当保险

### 语言信号 (用户的话里出现这些就应激活)

- "等融到B轮就好了"
- "投资人应该帮我们"
- "VC会救我们的"
- "我有投资人关系"
- "融资是创业的正常路径"

### 与相邻 skill 的区分

- 与 `g11-default-alive` 的区别: 本skill强调"为什么不要依赖投资人",g11强调"如何判断公司存活"
- 与 `ce07-overfunding-trap` 的区别: 本skill关注依赖外部资金的心理偏差,ce07关注融资后的行为问题

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **问"如果投资人永远不来，你怎么办"**
   - 完成标准: 用户能给出"止血"方案而非继续找投资人

2. **计算真实的Default Alive时间**
   - 完成标准: 用户计算出当前burn rate下的生存时间，并设定4个月止血目标

3. **重新定义融资角色**
   - 完成标准: 用户理解"融资是加速器，不是救生艇"
   - 判停条件: 若用户已实现Default Alive，融资可以讨论

---

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 已实现Default Alive后讨论扩张融资——这是合理的
- 讨论如何与投资人沟通——那是执行细节
- 传统企业融资——本skill专属于创业场景

### 作者在书中警告的失败模式

- 认为自己"融到资就安全了"
- 不关心公司的"默认存活/死亡"状态
- 等待外部救援而非自我造血

### 如何避免风投依赖幻觉

1. **永远假设投资人不会来** — 把账上钱的时间表减半作为真实跑道
2. **先做"止血手术"** — 在找投资人之前，先证明你能Default Alive
3. **理解投资人的激励结构** — 他们要的是高倍数，不是你的生存
4. **把"融资"从"起点"移到"终点之后"** — 先盈利，再谈融资

### 容易混淆的邻近方法论

- "融资能力是创业能力的一部分"——本skill不否定这个说法，而是强调优先级
- "与投资人建立关系"——关系可以有，但不能让关系替代执行力
