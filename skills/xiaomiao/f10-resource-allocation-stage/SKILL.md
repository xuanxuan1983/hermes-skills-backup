# 发展阶段资源配置框架

## R — 原文 (Reading)

> "早期，时间充裕而金钱稀缺。中期，你有时间管理金钱，但你没有足够的时间。后期，你有钱购买别人的时间，但没有足够的时间。理解你的处境，可以帮助你做出更好的决策。"

— Sahil Lavignena, 第六章

---

## I — 方法论骨架 (Interpretation)

这个框架揭示了一个反直觉的资源配置逻辑：不同发展阶段，核心稀缺资源不同，配置策略也应不同。

**早期阶段**: 你的时间是充裕的（因为你在全职创业），但金钱稀缺。正确策略是用时间换钱——做自由职业、接咨询、兼职。这阶段最危险的是"为了省钱浪费最宝贵的时间"。

**中期阶段**: 你开始有收入，但发现时间不够用了。金钱有所改善但依然紧张，时间成了真正的瓶颈。正确策略是用钱换时间——买软件、外包重复任务、自动化。这个阶段最危险的是"为了省钱亲力亲为"。

**后期阶段**: 你有了稳定的盈利，但时间极度稀缺。金钱充裕到可以"购买别人的时间"。正确策略是用钱买精力——招聘团队、雇佣助手。这阶段最危险的是"招人过早导致固定成本膨胀"。

**核心洞察：资源瓶颈随阶段变化，配置策略必须同步变化。早期用时间换钱，中期用钱换时间，后期用钱买精力。**

---

## A1 — 书中的应用 (Past Application)

### 案例 1: Gumroad 创始人薪资的演变

- **问题**: 创始人不知道应该给自己发多少工资
- **方法论的使用**: Sahil 把创始人薪资与公司状态挂钩——早期拿36000美元（够生存），盈利后逐步提高到120000美元
- **结论**: 创始人的报酬不是"应得的"，而是"公司状态的反映"
- **结果**: 薪资调整透明化，员工理解公司的盈利压力

### 案例 2: 软件 vs 人力的决策

- **问题**: 团队面临重复性任务，不知道应该招人还是买软件
- **方法论的使用**: 优先用软件解决，软件解决不了才考虑人力；因为软件成本低、可扩展、不需要管理
- **结论**: 软件在规模化后边际成本趋零，人力是线性成本且有管理成本
- **结果**: Gumroad 用软件工具构建了无会议、无办公室的异步文化

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill？

1. 用户月收入从0增长到几万，感到越来越忙但效率没有提升
2. 用户在考虑"要不要招第一个员工"，犹豫不决
3. 用户收到一笔钱（收入/融资/储蓄），在考虑怎么花
4. 用户抱怨"我每天工作12小时还是做不完"
5. 用户在规划年度预算，不知道应该投入什么

### 语言信号 (用户的话里出现这些就应激活)

- "我要不要招人"
- "这笔钱应该花吗"
- "怎么分配我的时间和金钱"
- "什么时候扩大团队"
- "我太忙了"
- "每月收入 X 万，但还是很累"

### 与相邻 skill 的区分

- 与 `p07-software-not-labor` 的区别: p07 是"软件优先于人力"的原则，f10 是"不同阶段用不同策略"的框架；p07 是 f10 的子决策
- 与 `p12-profitability-confidence` 的区别: p12 关注"要不要融资"，f10 关注"有资源后怎么分配"；两者都是资金决策，但层次不同

## E — 可执行步骤 (Execution)

当 skill 被激活后，agent 应按以下步骤执行:

1. **判断当前阶段**
   - 完成标准: 评估月净利润、每月工作时间、团队规模，确定处于早期/中期/后期
   - 问自己：我的瓶颈是"没客户"（早期）、"没时间"（中期）、还是"没精力"（后期）？

2. **确定资源配置策略**
   - 完成标准: 根据阶段选择正确的资源交换方向
   - 早期: 用时间换钱（优先做收入，克制做产品）
   - 中期: 用钱换时间（买软件、外包）
   - 后期: 用钱买精力（招聘、团队）

3. **评估具体决策**
   - 完成标准: 用框架评估当前面临的具体决策（招人/买工具/外包）
   - 问自己：这一步解决的是我当前的瓶颈吗？

4. **检查决策的一致性**
   - 完成标准: 确认你的决策符合当前阶段的策略
   - 判停条件: 若你处于早期却在考虑招全职员工，回到步骤1重新评估

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 用户处于收入为0的早期探索阶段——此时应该用 f01/f02 验证想法，不是资源配置
- 用户已有稳定盈利模型且规模较大——此时应该用更复杂的财务管理框架
- 用户的问题是"如何提高产品质量"——这是产品决策，不是资源配置决策

### 作者在书中警告的失败模式

- "在早期招人是自杀。固定成本会在你还没收入时压垮你"
- "为了省钱亲力亲为是另一种浪费——你把最宝贵的时间用在了最不值钱的工作上"

### 作者的盲点 / 时代局限

- 框架假设"时间"和"金钱"是主要资源，可能低估了"注意力"和"精力"作为独立资源的重要性
- 对于高利润业务（如 SaaS），金钱可能比时间更早充裕，阶段的边界可能不同

### 容易混淆的邻近方法论

- 精益创业的资源观: 精益强调"用最少资源验证"，这个框架强调"不同阶段用不同资源策略"
- Y Combinator 的 advice "做少而精的事": 两者都强调克制，但 YC 更偏向产品决策，这个框架更偏向资源配置
