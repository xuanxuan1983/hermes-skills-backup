# 反例: Interintellect的两次失败产品

## R — 原文 (Reading)

> 两年之内...很多曾表示会使用该软件的人并不像他们在初步调研中所表示的那样感兴趣。

> — 萨希尔·拉文吉亚, 第三章

---

## I — 方法论骨架 (Interpretation)

这个反例揭示了**口头调研与真实付费意愿之间的巨大鸿沟**。

核心逻辑链:
1. 用户说"会用" → 创始人认为"有需求"
2. 有需求 → 创始人开始开发
3. 开发完成后 → 用户不付费
4. 用户不付费 → 产品死亡

反直觉教训: 人们对"未来的自己会做什么"的预测极不准确。口头承诺≠付费意愿,热情≠需求。

---

## A1 — 书中的应用 (Past Application)

### 案例 1: Interintellect第一个产品
- **问题**: 社区合作进程比预期慢，产品无法扩张
- **方法论的使用**: 用"用户说会支持"判断产品方向
- **结论**: 口头承诺无法转化为实际参与
- **结果**: 产品失败

### 案例 2: Interintellect第二个产品
- **问题**: 用户说"会使用"，但两年后实际不感兴趣
- **方法论的使用**: 用初步调研决定开发方向
- **结论**: 调研热情与实际付费差距巨大
- **结果**: 产品再次失败

---

## A2 — 触发场景 (Future Trigger) ★

### 用户会在什么情境下需要这个 skill?

1. 用户做完调研后问"这个数据可信吗"——需要对调研结果进行验证
2. 用户说"80%表示会用"——把口头承诺当付费承诺
3. 用户在Reddit/Discord/问卷星上做调研——用网络问卷代替真实验证
4. 用户说"朋友都说好"——用社交圈代替目标市场
5. 用户说"我可以远程解决这个问题"——以为线上调研足够

### 语言信号 (用户的话里出现这些就应激活)

- "用户说他们会喜欢"
- "市场调研显示需求"
- "80%表示会使用"
- "朋友都觉得这个好"
- "我可以远程解决问题"

### 与相邻 skill 的区分

- 与 `g02-manual-valuable-process` 的区别: 本skill是"调研为什么不可靠",g02是"如何用人工方式验证"
- 与 `p04-零价格效应` 的区别: 本skill强调口头承诺的普遍不可靠,p04强调免费≠真实需求

---

## E — 可执行步骤 (Execution)

当 skill 被激活后, agent 应按以下步骤执行:

1. **测试"妈咪测试"**
   - 完成标准: 用户能把调研问题转换成"你愿意每月付X元解决Y问题吗"

2. **检查调研是否在已经建立信任的社区里**
   - 完成标准: 如果不是，问用户"这些人和你有信任关系吗"
   - 判停条件: 若调研来自强信任社区，可降低权重但仍需验证

3. **用付费行为验证**
   - 完成标准: 找到让用户付费的方法（哪怕1美元）——真金白银比口头更有说服力

---

## B — 边界 (Boundary) ★

### 不要在以下情况使用此 skill

- 已有真实付费用户的场景——这已经通过了最严格的验证
- 基于真实使用行为的数据（如Google Analytics）——行为数据比调研可靠
- 快速原型测试（A/B测试）——这是真实行为，不是口头承诺

### 作者在书中警告的失败模式

- 低估了社区建立所需时间
- 相信了用户"会使用"的口头承诺，而非付费验证
- 用热情代替了真实需求

### 如何避免 Interintellect 式的失败

1. **在调研时就让他们付费** — 哪怕1美元，愿意付的人才是真实用户
2. **用"妈咪测试"验证** — 问你妈妈（完全不懂你产品的人）愿不愿意付钱
3. **区分"喜欢"和"愿意付钱"** — 如果第一反应是"这值多少钱"而不是"当然"，那信号是假的
4. **口头调研只在强信任社区才相对可靠** — Reddit陌生人的"会用"和朋友圈的"会用"完全不同

### 容易混淆的邻近方法论

- "精益画布"中的"问题验证"——画布验证可以用口头数据，但需谨慎
- "用户访谈"——访谈是好的，但不能代替付费验证
